This is the full documented history across three physical build attempts, compiled from live diagnostic sessions since approximately June 1, 2026. Round trip to the builder costs ~6 hours and ~$60 in fuel each time, against a ~$700 total machine cost — so time and money have both been real constraints in how far this has been chased down.

Board A — The Original Build (~June 1–7/8, 2026)

Crashed repeatedly over 2-3 weeks. RAM chips were swapped out (2 removed, beeping stopped temporarily; later both replaced with new ones — crashed again within 2-3 hours). Conclusion at the time: the motherboard itself was faulty. Board A was returned to the builder.

Board B — Genuinely Different, Rebuilt Board

Not the same physical board as A. Builder didn't have a matching 1TB SSD, so Ryan's original SSD was moved from Board A into Board B, drivers updated, returned. Board B developed its own problems, including one bad USB port. Instability got bad enough that a Windows reinstall attempt could not complete — repeatedly stalling around 90%, requiring repair, crashing mid-install. That kind of failure is a strong sign of active hardware-level data corruption, not ordinary software flakiness.

Board B, New SSD — The Current Machine (as of 2026-07-01)

After Board B kept failing even with RAM and the whole board already changed, the SSD was the one component never yet swapped. Builder swapped in a new/different drive, same physical Board B. Machine crashed within about 2 hours of getting it home. Within 24 hours of pickup, it had crashed at least 4 times. It also developed a second bad USB port (on the same physical hub/controller as the first).

Why this points at the board itself, not the SSD: problems persisted across both the old and new drive, on the same board — a clean elimination. What's constant across every Board B failure, regardless of SSD, is the board (or the RAM/CPU/PSU paired with it).

2026-07-01 — Deep Diagnostic Session

Ruled out / confirmed healthy

  • AGESA/BIOS current (B.00, ComboAM4v2PI 1.2.0.F) — past the known-bad memory-instability range
  • CPU thermal: 60°C after 3 FurMark torture runs — normal
  • Chipset thermal: 40°C under load — normal for this test (though this board model has a documented separate overheating issue in the community)
  • GPU: clean FurMark runs, 83°C edge / ~105°C hotspot, no artifacts
  • SSD: ruled out via the natural swap-test (problem persisted across two different drives)

Still open / leading suspects at the time

  • RAM — G.Skill F4-3600C18-8GVK kit, running at conservative JEDEC 2133 MT/s (not even at its rated 3600 XMP speed) and still associated with crashes. This exact kit has documented MEMORY_MANAGEMENT/WATCHDOG_VIOLATION reports from other users. Running 4 DIMMs instead of 2 is also a known stability risk factor.

2026-07-03 — Older, More Rigorous Log Found: GPU Implicated Directly

A much more detailed prior investigation (Hardware_Troubleshooting_Log.md, covering ~May 27 – June 22, 2026, on a machine named DESKTOP-0AMCSMM) was found — compiled by an earlier session directly from session transcripts. It reaches a different primary conclusion: the used RTX 3070 GPU (unknown history, present since delivery) is the root cause, not RAM, not the motherboard.

Evidence: nvlddmkm Event ID 153 (GPU self-reporting a hardware fault) and Event ID 13 (ILLEGAL_MME_METHOD — GPU's internal command processor executing invalid instructions), surviving 3-4 clean driver reinstalls, normal temperatures throughout (68-72°C). Hypothesis: a failing GPU sharing the PCIe bus can generate electrical noise that Windows misreads as RAM/memory-controller errors — meaning the original "bad RAM" conclusions on Board A could actually trace back to a GPU problem instead.

Critical fingerprint match: exact same RAM part number, exact same BIOS version/date, same board, same CPU between that older document and the current machine. Whether it's the literal same physical GPU card across all three builds is unresolved — Windows exposes no true hardware serial for consumer GPUs; would require a physical label check inside the case.

A concrete live-caught data point (11:54:50 AM crash that day): new bugcheck type KERNEL_SECURITY_CHECK_FAILURE (0x139). Resource monitoring ruled out gradual memory pressure (RAM usage was trending down, not up, before the crash — a sudden, discrete event). Activity monitoring caught WerFault (Windows Error Reporting) firing ~71 seconds before the crash, then a second WerFault firing at the same moment NVIDIA Overlay started, 28 seconds before the full crash.

2026-07-05 — New Lead: A Failing Scheduled Backup Task

A Windows Scheduled Task named AmitClaudeBackup (created 2026-06-30, unrelated to the earlier board history) was found running every ~1 hour: robocopy "$env:USERPROFILE\.claude" "$env:USERPROFILE\OneDrive\Backups\.claude" /MIR /R:0 /W:0 — a full mirror copy of a live, constantly-growing session-log folder, with zero retry tolerance and set to fire immediately on every reboot to "catch up" on missed runs.

Correlation found: System log crash timestamps were compared directly against this behavior:

  • 7/3 morning: six reboots in ~2.5 hours (9:16, 9:45, 10:27, 10:57, 11:03, 11:18, 11:54 AM), each 20-38 minutes apart — a tight repeating-burst pattern consistent with "reboot triggers an immediate catch-up copy job, which collides with a live session and fails/hangs, contributing to the next crash."
  • 7/5: crashes at 4:06 PM and 5:03 PM, ~57 minutes apart — consistent with roughly-hourly retries.
  • 7/2: crash pairs at 3:19/3:25 PM and 4:05/4:08 PM — same tight-retry signature.

Mechanism: the live session transcript file is appended to continuously while any session is open. The backup task's zero-retry robocopy read hitting that file at the exact moment it's being written is a real, concrete cause of I/O contention — separate from, and possibly compounding, the GPU/RAM/board issue.

Action taken: task disabled 2026-07-05. Since it was created 2026-06-30 — after Board A's entire crash history and after Board B's earliest crashes (3:19 PM, 5:41 PM, 9:41 PM on day one) — it cannot explain the full three-board saga. It is judged a second, later-arriving, compounding cause layered on top of an already-unstable machine, not the sole explanation.

Where This Stands

Two live, unresolved threads, not mutually exclusive:

  • Hardware: GPU (used card, hardware-fault event codes) and/or RAM (documented crash reports on this exact part number, running 4-DIMM) and/or board VRM, on the same MSI B450M PRO-VDH MAX board across all three build attempts.
  • Software/task: the now-disabled hourly backup task, which correlates with several of the tightest recent crash clusters but cannot account for the earliest history.

Not yet done: MemTest86 (only the weaker Windows Memory Diagnostic has run, and it came back clean despite a crash 9 minutes later); physical confirmation of whether the GPU is the same card across all three builds.
